## Overview

Request contents are stored on IPFS. When privacy is required, requests can be encrypted so only authorized stakeholders can read them.

Request encryption combines:

- **Symmetric encryption (AES)** for request content
- **Asymmetric encryption** for sharing the symmetric key with authorized stakeholders

## Encryption flow

1. Generate a random AES key.
2. Encrypt request payload with AES.
3. Encrypt AES key for each stakeholder (payer, payee, approved third parties).
4. Persist encrypted payload and encrypted keys.

## Decryption flow

1. Retrieve encrypted payload and encrypted keys.
2. Decrypt AES key with stakeholder private key (or wallet-based flow).
3. Decrypt payload with AES key.

## Why this design

- Keeps sensitive business data private
- Preserves decentralized storage and interoperability
- Supports selective access for approved parties
